# Phase 1 — Port Congestion Dataset Design

Fully synthetic, generated by `src/generate_data.py`, seeded (`RANDOM_SEED = 42`) for
reproducibility. 500 rows, one row per inbound shipment. Regenerate with:

```
uv run python src/generate_data.py
```

## Schema

| Column                 | Type        | Range / values                              | Meaning                                                        |
| ---------------------- | ----------- | ------------------------------------------- | -------------------------------------------------------------- |
| `vessel_type`          | categorical | Container (50%) / Bulk (30%) / Tanker (20%) | vessel class                                                   |
| `weather`              | categorical | Clear (70%) / Rain (20%) / Storm (10%)      | conditions at arrival                                          |
| `queue_length`         | int         | 0–25                                        | vessels waiting ahead                                          |
| `port_utilization`     | float       | 60–100                                      | % of port capacity in use                                      |
| `historical_avg_delay` | float       | 0–12h                                       | route's recent delay history                                   |
| `arrival_hour`         | int         | 0–23                                        | hour of day — generated, **not yet used** in the delay formula |
| `delay_hours`          | float       | ≥ 0                                         | simulated total delay (latent, continuous ground truth)        |
| `is_delayed`           | int {0,1}   | —                                           | target: 1 if `delay_hours > 24`                                |

## How `delay_hours` is simulated

```
delay_hours = historical_avg_delay
            + weather_delay          (Clear=0, Rain=4, Storm=10)
            + vessel_delay           (Container=2, Bulk=4, Tanker=6)
            + 0.55 * queue_length * 0.8                              # queue_effect
            + 0.55 * (port_utilization - 60) * 0.25                  # utilization_effect
            + 0.55 * queue_length * (port_utilization / 100) * 0.4   # interaction_effect
            + noise ~ N(0, 2)
clipped at 0
```

Interpretation: baseline delay comes from the route's history plus flat penalties for
bad weather and slower vessel classes. On top of that, a longer queue and a busier port
each add delay independently, and — via `interaction_effect` — compound each other (a
long queue costs more when the port is already near capacity than when it's quiet).
`noise` represents unmodeled real-world variance.

## Why the 0.55 scale factor

At the raw formula (no scaling), mean `delay_hours` was ~30h — already above the 24h
threshold — so ~70% of rows came out "delayed," a majority-positive class. That
contradicts the 30–45% minority target in `planning.md` §2: real congestion should be
the exception, not the norm, and that imbalance is also _why_ AUC/F1 were chosen over
accuracy as eval metrics — an inverted-majority class would undercut that reasoning.

`OPERATIONAL_EFFECT_SCALE = 0.55` scales down only the three _operational_ terms
(queue, utilization, interaction) — found by simulating the delay rate at several scale
values and picking one that lands centrally in the target band:

| scale     | delay rate |
| --------- | ---------- |
| 1.0 (raw) | 70%        |
| 0.6       | 44%        |
| **0.55**  | **37%**    |
| 0.5       | 33%        |
| 0.3       | 12%        |

`DELAY_THRESHOLD = 24` was left untouched — it's the operationally meaningful boundary
(§2), not a knob to fit prevalence. Only how easily the formula crosses it changes.

## Ground truth definition

`is_delayed = 1` if `delay_hours > 24`, else `0`. See `planning.md` §2 for why 24h.

## Realism / limitations

- Single generic port modeled — no per-port variation.
- Effects are additive plus one pairwise interaction (queue × utilization); real
  congestion likely has higher-order interactions (e.g., weather × queue) not modeled.
- `arrival_hour` is generated but **not wired into the delay formula** — no
  time-of-day/seasonality effect is simulated, despite the brief flagging temporal
  dynamics as a suggested challenge. Documented as a known gap, not a bug.
- Noise is small and Gaussian (`std=2h`); real delay variance is likely fatter-tailed
  (mechanical failures, labor actions, customs holds), which this simulation doesn't capture.
- All coefficients (weather/vessel penalties, effect weights, the 0.55 scale) are
  hand-set to produce plausible relative magnitudes and the target class balance — not
  fit from real data. Reasonable for a synthetic prototype, but stated here explicitly
  as an assumption for the report.
